# AntFlowCore 虚拟节点(VNode)模式深度解析

## 前言

AntFlow 全网首创了**虚拟节点(VNode)** 模式，将流程流转业务和引擎执行 API 高度分离，让开发者不需要懂复杂的 BPMN 规范，也能轻松开发流程审批业务。

本文深入解析虚拟节点模式的设计思想、实现原理，以及为什么它能让流程开发变得如此简单。

## 什么是虚拟节点模式

### 传统流程引擎的问题

传统流程引擎（如 Activiti、jBPM）采用的是**物理节点**模式：

- 流程定义中的每一个节点都是物理存在的
- 开发者需要为每个节点编写 `TaskListener`、`ExecutionListener`
- 需要理解复杂的 BPMN 规范
- 需要学习引擎大量的 API
- 流程流转逻辑散落在各个监听器中，难以维护

举个例子，传统方式实现一个"部门经理审批"节点：

```java
// Activiti 传统方式
public class DepartmentManagerApprovalListener implements TaskListener {
    @Override
    public void notify(DelegateTask delegateTask) {
        // 1. 查询部门经理
        // 2. 设置审批人
        // 3. 发送通知
        // 复杂的 API 调用...
    }
}
```

每个节点都要写一个监听器，项目大了之后，几十个节点就有几十个监听器，非常零散。

### 虚拟节点模式的思路

**虚拟节点模式的核心思想：流程定义中的节点只是占位符，真正的流转逻辑由开发者通过一套统一的接口来实现。**

换句话说：

- BPMN 图只负责定义流程走向（谁到谁）
- 具体找哪个审批人、什么条件能过，全部交给你的业务代码通过统一方式处理
- 不需要为每个节点写单独的监听器
- 所有逻辑都在你自己的业务系统中，完全可控

### 举个例子理解虚拟节点

假设你有一个请假流程：

```
发起申请 → 部门经理审批 → 总经理审批 → 结束
```

在 BPMN 图中，"部门经理审批"就是一个虚拟节点，它只表示这里需要一个审批环节，具体：

- **谁是部门经理？** → 你自己的业务代码算
- **什么条件跳过？** → 你自己的业务代码算
- **审批通过后做什么？** → 你自己的业务代码处理

AntFlow 只负责帮你推进流程，保存状态，而把业务决定权完全交给你。

这是最核心的接口，AntFlow 走到一个虚拟节点时，会调用这个接口让你告诉它，这个节点应该由谁来审批。

### 2. IConditionJudge - 判断节点条件

```csharp
public interface IConditionJudge
{
      bool Judge(String nodeId, BpmnNodeConditionsConfBaseVo conditionsConf, BpmnStartConditionsVo bpmnStartConditionsVo,int coundGroup);
}
```

当流程走到排他网关或者条件节点时，会调用这个接口判断条件是否满足。

### 3. ITaskListener- 流程执行回调

```csharp
public interface ITaskListener
{
   const String EVENTNAME_CREATE = "create";
   const String EVENTNAME_RE_SUBMIT = "resubmit";

   const String EVENTNAME_DELETE = "delete";
   void Notify(BpmAfTask delegateTask,string eventName);
```

```
└────────────────────────────┬────────────────────────────────┘
                             │
                             ▼
┌─────────────────────────────────────────────────────────────┐

                             ▼

                             ▼
```

## VNode 模式的优势

### 1. 学习成本低

- 不需要理解复杂的 BPMN 规范细节
- 只需要学会两三个接口
- 会写业务代码就能开发流程

### 2. 高内聚，低耦合

- 所有审批人查找逻辑在一个类
- 所有条件判断逻辑在一个类
- 相比分散的监听器，更容易维护
- 你的业务逻辑完全在你自己的项目中

### 3. 高度灵活

- 可以复用现有的业务服务
- 可以随时修改规则，不需要重新部署流程定义
- 支持极其复杂的审批规则

### 4. 和你的业务系统深度集成

- 可以直接调用你的业务服务查询数据
- 不需要把用户、组织数据复制一份到流程引擎
- 永远保持数据一致性

## 和传统模式对比

| 对比项         | 传统物理节点模式                 | AntFlow 虚拟节点模式         |
| -------------- | -------------------------------- | ---------------------------- |
| 学习曲线       | 陡峭（需要学会 BPMN + 大量 API） | 平缓（只需两三个接口）       |
| 代码组织       | 分散（每个节点一个监听器）       | 集中（统一处理）             |
| 和业务系统集成 | 需要复制数据到引擎               | 直接调用业务服务             |
| 灵活性         | 低（规则修改往往需要重新部署）   | 高（规则在业务代码，随时改） |
| 维护成本       | 高（节点越多，类越多）           | 低（集中管理）               |

## 最佳实践建议

### 1. 按照职责分组

你可以进一步拆分实现，比如：

```
- 审批人查找
  - DepartmentManagerAssigneeProvider
  - RoleBasedAssigneeProvider
  - UserBasedAssigneeProvider
```

然后在 实现中根据节点配置分发调用。
